# Сценарий ручной проверки Telegram-бота EDme

Проверка проводится в реальном Telegram-чате с ботом. Сервер должен быть запущен
(`npm start`), вебхук указывает на актуальный туннель, `DATABASE_URL` — рабочая база.

Помечено **[AI]** — шаги, которые тратят реальные вызовы OpenRouter/OpenAI (платные).
Помечено **[МК]** — требует настроенного `MOYKLASS_API_KEY` и реального аккаунта в Мой Класс.
Помечено **[МТС]** — требует настроенного `MTS_LINK_API_TOKEN`/`MTS_LINK_WEBHOOK_SECRET`.

---

## 0. Подготовка

- [ ] `GET /health` возвращает `{"ok": true, "db": "up"}`.
- [ ] В базе есть хотя бы один репетитор с известным email/телефоном (демо-сид при
      `SEED_DEMO_DATA=true`, либо реальная запись из Мой Класс).
- [ ] Второй Telegram-аккаунт (или второй тестовый чат) для проверки сценариев
      конкурентного доступа — по желанию, не обязателен.

---

## 1. Авторизация и базовые команды

1.1. Отправить `/start` на чистом аккаунте (не привязанном ни к одному репетитору).
  - Ожидается: «Введите email или номер телефона…».

1.2. Ввести заведомо неверный email.
  - Ожидается: «Профиль не найден. Проверьте email или номер телефона… обратитесь к администратору.»
  - Сессия остаётся в состоянии ожидания ввода (можно ввести повторно).

1.3. Ввести верный email/телефон существующего репетитора.
  - Ожидается: главное меню, сообщение «Вы вошли как {имя}.», следом — попытка
    `refreshStudents` (если `MOYKLASS_API_KEY` не настроен — сообщение «Мой Класс не
    настроен на сервере», это нормально).

1.4. Отправить `/start` повторно уже авторизованным.
  - Ожидается: сразу главное меню, без повторного запроса идентификации.

1.5. `/menu`, `/students`, `/help`, `/settings` — каждая команда отвечает корректно
     авторизованному пользователю.
  - `/settings` показывает ФИО, email/телефон, Мой Класс ID (или «не связан»), кнопку
    «Сменить аккаунт».

1.6. **Повторное использование идентичности другим Telegram-аккаунтом.**
  - Со второго Telegram-аккаунта ввести email/телефон **уже привязанного** репетитора.
  - Ожидается: «Этот профиль уже привязан к другому Telegram-аккаунту. Обратитесь к
    администратору.» (это исправленный текст — раньше было перепутано с обратным случаем).

1.7. `/logout` (или кнопка «Сменить аккаунт»).
  - Ожидается: «Вы вышли из аккаунта.», затем сразу предложение ввести email/телефон
    заново. Повторный `/start` не должен считать пользователя авторизованным.

1.8. Проверить, что бот **не отвечает** во групповом чате (добавить бота в тестовую
     группу, написать текст).
  - Ожидается: полное молчание (это осознанное изменение — бот больше не отвечает вне
    приватных чатов).

1.9. Отправить `/` + опечатка (`/unknown`) — не должно ронять бота, TelegramBotAPI
     просто не найдёт обработчик.

## 2. Список учеников и пагинация

2.1. `/students` при отсутствии учеников (новый репетитор без Мой Класс).
  - Ожидается: «Пока нет учеников. Нажмите «Обновить учеников» или отправьте /refresh.»

2.2. **[МК]** `/refresh` (или кнопка «Обновить учеников») у репетитора с привязкой к
     Мой Класс.
  - Ожидается: «Обновляю учеников из Мой Класс по вашим занятиям…», затем «Готово:
    занятий N, учеников M, новых расшифровок K.»
  - Проверить, что синк укладывается в разумное время (несколько секунд/десятков секунд,
    не минуты) — это должно быть заметно быстрее, чем раньше, благодаря окну дат
    180 дней назад / 60 вперёд.

2.3. Нажать «Мои ученики» — список должен показать кнопки по каждому ученику
     `Имя — Предмет, N класс`.

2.4. **Пагинация списка учеников** (нужно ≥31 ученик у репетитора, либо временно
     понизить `STUDENTS_PAGE_SIZE` в `src/telegram.js` для теста и вернуть обратно).
  - Ожидается: после 30 кнопок — кнопка «Показать ещё (N)», по нажатию подгружается
    следующая страница.

## 3. Карточка ученика

3.1. Выбрать ученика из списка → открывается карточка с меню действий.
  - Проверить, что в карточке отображается блок **«Ближайшее занятие»** (новое поле) —
    либо дата, либо «не запланировано».

3.2. Нажать «Карточка» — повторный вывод карточки без кнопок действий ученика.

3.3. **Редактирование карточки вручную** — «Редактировать карточку»:
  - Пройти по каждому полю: Уровень, Сильные стороны, Слабые стороны, Пробелы,
    Рекомендации, Цель обучения, Темп обучения, Комментарии, План следующего урока.
  - Для списковых полей (сильные/слабые стороны, пробелы, рекомендации) — ввести
    несколько строк, проверить, что каждая стала отдельным пунктом.
  - Убедиться, что после отправки бот показывает обновлённое значение поля.
  - Открыть карточку заново — изменения должны сохраниться.

3.4. **Устаревшая (протухшая) сессия редактирования.**
  - Начать редактирование поля («Отправьте новое значение…»), подождать дольше TTL
    сессии (по умолчанию 30 минут — для теста можно временно уменьшить
    `DEFAULT_TTL_SECONDS` в `src/telegram/session-store.js`), затем отправить текст.
  - Ожидается: «Время ожидания истекло, действие отменено. Откройте /menu и начните
    заново.» — сообщение НЕ должно уйти в свободный ИИ-чат.

3.5. «Последние темы» — список последних занятий из Мой Класс (или «Занятий пока не
     найдено…», если МК не настроен/нет данных).
  - **[МК]** Пагинация: если у ученика больше 4 занятий — кнопка «Показать раньше».

## 4. Генерация ДЗ / теста / задач [AI]

Для этого раздела нужен настроенный `OPENROUTER_API_KEY` (или `OPENAI_API_KEY`).

4.1. **Домашнее задание.**
  - Карточка ученика → «Создать ДЗ» → ввести тему («Квадратные уравнения»).
  - Ожидается: сообщение «Генерирую домашнее задание…», затем структурированный
    результат (Ученик/Предмет/Класс/Тема/Цель/Задания/Доп. задание/Комментарий) и
    клавиатура с кнопками: 🔄 Заново, Проще, Сложнее, Короче, Больше практики,
    ✏️ Своя правка, 👍/👎.
  - Если у ученика в карточке заполнены «Пробелы» — проверить, что в тексте ДЗ есть
    отметка про их учёт.
  - Повторить с темой «Сложение и вычитание дробей». Проверить, что `3/4`, `1/8`
    и смешанные числа отображаются как вертикальные LaTeX-дроби, а символы `$`,
    `\frac`, `#` и `**` не видны пользователю.

4.2. **Тест.**
  - «Создать тест» → тема.
  - Ожидается: тест с вариантами ответов + раздел «Ответы для репетитора» в конце.
  - Проверить дополнительные кнопки только для теста: «Больше вопросов», «Меньше
    вопросов», «Версия для ученика».
  - Нажать «Версия для ученика» — убедиться, что раздел с ответами обрезан
    (`studentVersion`).

4.3. **Задачи разного уровня.**
  - «Сгенерировать задачи» → тема.
  - Ожидается: три уровня (Базовый/Средний/Повышенный) + рекомендация с какого начать.
  - Проверить кнопку «Решения и ответы» (только для типа `tasks`) — по нажатию должен
    прийти отдельный текст с решениями/подсказками ко всем заданиям.

4.4. **Корректировки генерации** — на любой из трёх типов:
  - 🔄 Заново — новый вариант по той же теме.
  - Проще / Сложнее — заметно меняется сложность, тема сохраняется.
  - Короче — объём уменьшается.
  - Больше практики — прибавляется практических заданий.
  - ✏️ Своя правка — бот просит: «Опишите одним сообщением, что изменить…», ввести,
    например, «замени вторую задачу на задачу про проценты» → должен прийти
    скорректированный вариант.

4.5. **Обратная связь по генерации.**
  - 👍 Подходит — «Спасибо! Отметил, что материал подходит.»
  - 👎 Не подходит — появляются кнопки причин (Не по теме / Ошибка в задании / Слишком
    много / Слишком мало) → выбрать одну → подтверждение сохранения причины.

4.6. **Rate limit / busy guard.**
  - Запустить генерацию (шаг 4.1) и **сразу**, не дожидаясь ответа, нажать ещё одну
    кнопку генерации (или отправить новую тему).
  - Ожидается: «Предыдущий запрос к ИИ ещё выполняется. Дождитесь результата.» —
    вторая генерация не должна запуститься параллельно (тратятся деньги/путается state).
  - Проверить, что после завершения первой генерации кнопки снова доступны.
  - (Опционально, для лимита в 10/мин) быстро наштамповать >10 ИИ-действий подряд —
    должно появиться «Слишком много запросов подряд. Подождите минуту…».

4.7. **Двойное нажатие кнопки (double-tap) в целом.**
  - На любой ИИ-кнопке кликнуть дважды подряд очень быстро.
  - Ожидается: второй клик либо блокируется busy-guard'ом (см. 4.6), либо, если первый
    уже завершился — создаёт отдельную корректную генерацию. Не должно быть зависшего
    спиннера на кнопке дольше пары секунд (answerCallbackQuery теперь отвечает сразу).

4.8. **ИИ не настроен.**
  - Временно очистить `OPENROUTER_API_KEY`/`OPENAI_API_KEY` в `.env`, перезапустить
    сервер, попробовать «Создать ДЗ».
  - Ожидается: «ИИ недоступен: настройте ключ ИИ на сервере.» — без падения бота.
  - Вернуть ключ и перезапустить сервер перед следующими шагами.

## 5. Поиск материалов [МТС]

Нужно, чтобы `/api/materials/sync` был выполнен хотя бы раз (см. п. 8) и в базе
были материалы.

5.1. Карточка ученика → «Найти материал» → ввести тему, например «квадратные
     уравнения».
  - Ожидается: список найденных материалов с названием/типом/уровнем/ссылкой.

5.2. Ввести запрос с классом внутри текста: «квадратные уравнения 8 класс».
  - Ожидается: класс «8» должен восприниматься как фильтр по классу (а не как
    текстовый токен) — результаты не должны теряться из-за буквального совпадения
    слова «класс» в названии.

5.3. Запрос, не совпадающий ни с чем: «несуществующая тема xyz123».
  - Ожидается: «Ничего не найдено. Попробуйте другой запрос.»

5.4. Запрос по типу материала — «тест по теме X» — должен находить материалы с
     `material_type` похожим на «тест» (тестовый охват токенизированного поиска).

## 6. Расшифровки и авто-анализ урока

6.1. **Добавление расшифровки вручную.**
  - Карточка ученика → «Добавить расшифровку» → ввести короткий текст (< 20 символов,
    например «ок»).
  - Ожидается: «Слишком короткий текст для расшифровки…» — расшифровка не создаётся.
  - Повторить с текстом длиннее 20 символов, например: «Разобрали тему квадратных
    уравнений, ученик путался в дискриминанте.»
  - Ожидается: «Расшифровка сохранена. Откройте «Расшифровки»…»

6.2. «Расшифровки» у ученика → должен появиться список с датой и статусом (`new`).
  - **Пагинация:** если расшифровок больше 10 — кнопка «Показать ещё».

6.3. Открыть расшифровку (нажать на неё) → показывается текст (обрезанный, если
     длиннее 3500 символов) + кнопка «Анализировать».

6.4. **[AI] Анализ расшифровки.**
  - Нажать «Анализировать».
  - Ожидается: «Анализирую расшифровку урока. Это может занять до минуты.», затем
    черновик с блоками: Тема, Что получилось, Трудности, Пробелы, Рекомендации, План
    следующего урока — и клавиатура:
    - ✅ Подтвердить
    - ✏️ Пробелы / ✏️ Рекомендации
    - ✏️ Что получилось / ✏️ Трудности
    - ✏️ План урока

6.5. **Редактирование черновика (закрывает критерий MVP №11).**
  - Нажать любую кнопку «✏️ …», ввести новое значение (для списков — по строке на
    пункт).
  - Ожидается: черновик присылается заново с обновлённым значением и той же
    клавиатурой (можно редактировать несколько полей подряд, до подтверждения).

6.6. **Подтверждение черновика.**
  - Нажать «✅ Подтвердить».
  - Ожидается: «Карточка обновлена. Изменение сохранено в истории.»
  - Открыть карточку ученика → отредактированные значения должны попасть в карточку
    (Пробелы/Рекомендации/Что получилось→Сильные стороны/Трудности→Слабые стороны/План).
  - Повторное нажатие «✅ Подтвердить» на том же черновике → «Черновик уже обработан
    или не найден.» (защита от повторного подтверждения).

6.7. **Анализ при отвязанном ученике.**
  - (Требует ручной манипуляции в БД или отвязки ученика от репетитора.) Попробовать
    анализировать расшифровку ученика, который больше не привязан к репетитору.
  - Ожидается: «Ученик не найден или больше не привязан к вам.» — без падения бота.

## 7. Свободный вопрос (freeform) [AI]

7.1. Находясь в главном меню (без активной сессии ввода), написать произвольный
     вопрос: «Как объяснить восьмикласснику формулу дискриминанта простыми словами?»
  - Ожидается: «Готовлю ответ. Это может занять до минуты.», затем содержательный
    ответ с отрендеренными заголовками и формулами. Исходные маркеры Rich Markdown
    и LaTeX пользователю не видны.

7.2. Не авторизованный пользователь пишет произвольный текст.
  - Ожидается: «Сначала выполните /start и авторизацию.»

7.3. Повторить проверку rate-limit/busy guard (см. 4.6) применительно к freeform —
     быстро отправить два вопроса подряд.

## 8. HTTP API / интеграции (не Telegram, но часть системы)

Использовать `curl`/Postman с заголовком `x-api-secret: <INTERNAL_API_SECRET>`.

8.1. `GET /health` без заголовков — должен быть доступен без секрета, отдавать
     `db: "up"` при живой БД. Остановить БД (или испортить `DATABASE_URL` временно) —
     должен вернуть `503` с `db: "down"`, сервис не должен упасть.

8.2. Любой `/api/...` без `x-api-secret` → `401`. С неверным секретом → `401`. С верным
     → проходит дальше.

8.3. `POST /api/moy-klass/sync` **[МК]** — полная синхронизация компании. Проверить,
     что в ответе приходит `{tutors, students, tutorStudentLinks}`.

8.4. `POST /api/materials/sync` **[МТС]** — синк материалов из МТС Линк. Проверить
     ответ `{materials: N}` и что после этого поиск в п.5 находит записи.

8.5. `GET /api/students?tutorId=...` и `GET /api/students/:id?tutorId=...` — базовая
     выдача, включая `card`.

8.6. `PATCH /api/students/:id/card` — ручное обновление через API, проверить, что
     конкурентная правка с ботом не теряет данные (шаг 9.2).

8.7. `GET /api/metrics` — проверить, что возвращаются реальные счётчики использования
     (генерации по типам, доля перегенераций, фидбек, аналитика по расшифровкам).

8.8. `POST /webhooks/mts-link` **[МТС]** — прогнать тестовое событие из личного
     кабинета МТС Линк (или руками с правильной HMAC-подписью). Проверить в логах
     сервера, что подпись валидируется и `processTranscriptReady` отрабатывает.
  - Негативный кейс: отправить запрос без подписи / с неверной подписью → `401`.

8.9. `POST /webhooks/telegram` без заголовка `x-telegram-bot-api-secret-token` (или
     с неверным) → `401`. Настоящие апдейты от Telegram (уже проверено всем разделом 1-7)
     обязаны идти с верным секретом автоматически.

## 9. Параллельность и устойчивость

9.1. **Конкурентное редактирование карточки.**
  - Открыть редактирование одного и того же поля карточки одновременно в боте и через
    `PATCH /api/students/:id/card`, отправить оба изменения почти одновременно.
  - Ожидается: оба изменения применяются последовательно (никакое не «теряется» —
    сейчас каждое обновление блокирует строку `FOR UPDATE` в транзакции), в
    `card_history` должно быть две записи.

9.2. **Устойчивость к ошибкам без падения процесса.**
  - Временно испортить `DATABASE_URL` (или остановить БД) и отправить команду
    в Telegram, например `/students`.
  - Ожидается: бот отвечает разумным сообщением об ошибке («Не удалось обработать…»),
    процесс сервера **не падает** (проверить `GET /health` после этого — сервис должен
    быть жив и отвечать, хоть и с `db: down`).
  - Восстановить `DATABASE_URL`/БД, повторить `/students` — должно снова заработать.

9.3. **Graceful shutdown.**
  - Отправить процессу сервера `SIGTERM` (или `Ctrl+C` в терминале, где запущен
    `npm start`) во время обработки запроса.
  - Ожидается в логах: «SIGTERM received, shutting down», корректное закрытие
    HTTP-сервера и пула БД, процесс завершается сам (без зависания на 15 сек. таймауте).

9.4. **Локальный polling fallback.**
  - Убрать `TELEGRAM_WEBHOOK_URL` из `.env` (или выставить `TELEGRAM_POLLING=true`),
    перезапустить сервер.
  - Ожидается в логах: «Telegram polling started…». Бот должен продолжать отвечать на
    сообщения без публичного URL/туннеля.
  - Вернуть `TELEGRAM_WEBHOOK_URL`, перезапустить — должно вернуться «Telegram webhook
    configured».

## Быстрый чек-лист «за 15 минут» (дымовой прогон)

Если нет времени на полный сценарий — минимальный набор, покрывающий все критичные
фиксы этой итерации:

1. `/start` → авторизация → главное меню.
2. `/students` → выбрать ученика → карточка (проверить «Ближайшее занятие»).
3. Создать ДЗ по теме → дождаться результата → нажать 👍.
4. Сразу после генерации нажать «Проще» **и** мгновенно «Сложнее» вторым тапом —
   убедиться, что второе действие блокируется busy-guard'ом, а не запускает вторую
   параллельную генерацию.
5. Добавить расшифровку → «Анализировать» → отредактировать поле «Пробелы» через
   ✏️-кнопку → «✅ Подтвердить» → открыть карточку, убедиться, что пробелы обновились.
6. Найти материал по теме с упоминанием класса («... 8 класс») — убедиться, что поиск
   не ломается на слове «класс».
7. Временно погасить БД → отправить `/menu` → бот должен ответить ошибкой, не упасть
   (проверить `curl .../health` — сервер по-прежнему жив).
8. `/logout` → повторный `/start` → снова запрашивает идентификацию.
